# game-prd-writer — MVP 기반 게임 PRD 작성

## 목적

게임 분석 결과 또는 아이디어를 받아 **실행 가능한 PRD**를 작성합니다.

PRD는 **WHAT(무엇을)과 WHY(왜)**를 정의하는 문서입니다.
- ✅ 포함: 기능 명세, 수용 기준, 성공 지표, 단계별 범위
- ❌ 제외: 구현 방법, 기술 아키텍처, 개발 일정, 코드 구조

---

## 입력 형태

다음 중 하나를 입력받습니다:

- **게임 분석 결과**: `game-analysis` 스킬이 작성한 분석 문서 (MDA, 루프, Octalysis 포함)
- **게임 아이디어**: 만들고 싶은 게임 설명 (장르, 핵심 메카닉, 대상 플레이어 등)
- **두 가지 조합**: 아이디어 + 레퍼런스 게임 분석

입력이 부족하면 다음을 질문합니다:
1. 플레이어가 가장 재미있을 것 같은 순간 (한 문장)
2. 대상 플레이어 (타겟 유저층)
3. 타겟 플랫폼 (PC / 모바일 / 콘솔)
4. 팀 규모

---

## PRD 작성 워크플로우

### 1단계: 핵심 경험 추출

`game-analysis` 결과가 있으면 **에스테틱 → 다이나믹스 → 메카닉** 역방향으로 핵심 경험을 도출합니다.

```
에스테틱 (게임 분석의 MDA 분석 섹션에서):
  → 이 게임이 플레이어에게 주는 감정이 무엇인가?

다이나믹스 (게임 분석의 핵심 루프 섹션에서):
  → 그 감정을 만드는 반복 행동 패턴은 무엇인가?

핵심 경험 선언문:
  → "플레이어가 [행동]을 반복하면서 [감정]을 경험하는 게임"
```

분석 결과 없이 아이디어만 있을 경우:
- "이 게임의 가장 재미있는 순간을 한 문장으로 설명하면?" 질문
- 그 순간을 만드는 최소 조건을 역산

### 2단계: MVP 범위 산정

핵심 경험을 느끼기 위한 **최소 필요 기능**만 포함합니다.

**포함 기준**:
- ✅ 핵심 경험을 직접 구성하는 기능
- ✅ 핵심 경험을 느끼기 위해 없으면 불가능한 기능
- ❌ "있으면 좋은" 기능 → 알파 이후로
- ❌ 콘텐츠 다양성 (1가지로 검증 가능) → 알파 이후로
- ❌ 완성도 (그래픽, 사운드, 애니메이션) → 베타 이후로
- ❌ 메타 시스템 (세이브, 업적, 랭킹) → 알파 이후로

**MVP 범위 체크**:
```
□ 플레이어가 핵심 행동을 1회 수행할 수 있는가?
□ 그 행동에 의미 있는 피드백이 있는가?
□ 승리/실패 또는 진행/종료 조건이 있는가?
□ 핵심 루프를 3회 이상 반복할 수 있는가?
□ 재시작(또는 다음 시도)이 가능한가?
```

### 3단계: 기능 명세 작성

각 기능을 다음 형식으로 작성합니다:
- **설명**: 이 기능이 무엇인지 (WHAT)
- **플레이어 시나리오**: "플레이어는 ~할 수 있다"
- **수용 기준**: 기능이 올바르게 동작함을 판단하는 구체적 조건

### 4단계: 단계별 범위 정의

MVP → 알파 → 베타 → 출시 준비 순으로 기능을 배치합니다.
각 단계는 이전 단계 위에 쌓이며, 완료 시 플레이 가능한 빌드가 존재해야 합니다.

### 5단계: 리스크 식별

핵심 경험 가설, 기술적 불확실성, 범위 팽창 위험을 식별합니다.

### 핵심 경험 선언문

> "플레이어가 [핵심 행동]을 반복하면서 [핵심 감정/에스테틱]을 경험하는 게임"

**MVP가 검증해야 하는 가설**:
- 가설 1: [핵심 경험 가설]
- 가설 2: [보조 가설 (있는 경우)]

**MVP 성공 기준** (플레이 테스트 기준):
- [ ] [정량적 기준 1] — 예: "처음 해보는 플레이어가 5분 이내에 핵심 루프를 스스로 파악한다"
- [ ] [정량적 기준 2] — 예: "플레이어 2명 중 1명 이상이 자발적으로 재시작한다"
- [ ] [정성적 기준] — 예: "플레이 후 '한 번 더 하고 싶다'는 반응이 나온다"

- **단계**: MVP 필수
- **설명**: [이 기능이 무엇인지, 1~2문장]
- **플레이어 시나리오**: "플레이어는 [행동]을 할 수 있다. [결과]가 발생한다."
- **수용 기준**:
  - [ ] [구체적 조건 1]
  - [ ] [구체적 조건 2]

### 알파 기능 명세

**알파 목표**: 전체 게임 루프 완성 + 콘텐츠 다양성 확보 + 내부 플레이 테스트

**MVP → 알파 전환 조건**:
- [ ] MVP 플레이 테스트에서 핵심 경험 가설이 검증됨
- [ ] 외부인 3명 이상이 자발적으로 재시작함
- [ ] 치명적 버그(크래시, 진행 불가) 없음

#### 기능: [기능명]

- **단계**: 알파
- **설명**: [설명]
- **플레이어 시나리오**: [시나리오]
- **수용 기준**:
  - [ ] [조건]

**알파 완료 기준**:
- [ ] 전체 게임 루프를 처음부터 끝까지 플레이 가능
- [ ] 계획된 콘텐츠 70% 이상 구현
- [ ] 내부 플레이 테스트 10회 이상

---

### 베타 기능 명세

**베타 목표**: 전체 콘텐츠 완성 + 완성도 향상 + 외부 플레이 테스트

**알파 → 베타 전환 조건**:
- [ ] 전체 게임 루프 완성
- [ ] 난이도 곡선이 플레이 테스트에서 검증됨
- [ ] 알려진 크래시 버그 0건

#### 완성도 기준

| 영역 | 베타 목표 수준 |
|------|--------------|
| UI/UX | [목표 수준] |
| 사운드 | [목표 수준] |
| 비주얼 이펙트 | [목표 수준] |
| 성능 | [목표 수치 — PC: 60fps / 모바일: 30fps] |

**베타 완료 기준**:
- [ ] 외부 플레이 테스터 10명 이상 완주
- [ ] 크리티컬 버그 0건, 마이너 버그 10건 이하
- [ ] 목표 플랫폼에서 목표 프레임레이트 달성

### 출시 준비 요구사항

**출시 준비 목표**: 스토어 등록 가능 상태

**베타 → 출시 준비 전환 조건**:
- [ ] 외부 플레이 테스터 10명 이상 완주
- [ ] 모든 크리티컬 버그 해소
- [ ] 목표 플랫폼 기기에서 성능 목표 달성

#### 필수 요구사항

```
스토어/배포:
□ 스토어 페이지 자료 (스크린샷, 설명, 아이콘, 트레일러)
□ 개인정보 처리방침
□ 연령 등급 심사
□ [플랫폼별 추가 요구사항]

기술:
□ 크래시 리포팅 연동
□ 분석 도구 연동 (리텐션, 세션 시간 측정)
□ 원격 설정 (밸런스 수치 조정 가능)
```

**범위 조정 원칙**:
- 일정 압박 시: 콘텐츠 수를 줄이되 핵심 경험 품질 유지
- MVP에서 재미를 못 느끼면: 범위 확대 전에 핵심 경험 선언문 재검토
- 알파 이후 핵심 메카닉 변경: 신규 프로젝트 수준의 결정으로 간주

## 모범 사례

1. **핵심 경험 먼저**: 모든 기능은 "이게 없으면 핵심 경험을 느낄 수 없나?"로 필요성을 판단
2. **PRD는 WHAT과 WHY**: 구현 방법(HOW)은 PRD에 포함하지 않음 — 개발 단계에서 결정
3. **수용 기준은 검증 가능하게**: "잘 작동한다" 대신 "플레이어가 3초 이내에 결과를 인지한다"
4. **의도적 제외를 명시**: 포함하지 않는 것도 결정이며, 이유를 기록해야 범위 팽창을 방지
5. **게임 분석 출력 활용**: game-analysis의 에스테틱이 핵심 경험 선언문이 되고, 코어 루프가 MVP 범위의 기준이 됨
6. **플레이 테스트 조기 실시**: MVP 완료 즉시 외부인에게 플레이 테스트 — 가정을 데이터로 검증
7. **단계 전환 조건 준수**: 이전 단계 완료 기준을 충족하지 않으면 다음 단계로 넘어가지 않음
