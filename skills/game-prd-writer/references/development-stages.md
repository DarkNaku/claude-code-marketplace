# 단계별 전환 기준 및 전략

MVP → 알파 → 베타 → 출시 준비 각 단계의 전환 조건, 집중 전략, 플레이 테스트 방법 가이드.

---

## 단계별 개요

| 단계 | 핵심 질문 | 목표 빌드 상태 |
|------|-----------|--------------|
| **MVP** | "핵심 경험이 실제로 재미있는가?" | 플레이 가능한 프로토타입 |
| **알파** | "전체 게임 루프가 완성되었는가?" | 내부 플레이 가능 빌드 |
| **베타** | "출시할 수 있는 수준의 완성도인가?" | 외부 테스트 빌드 |
| **출시 준비** | "스토어에 올릴 수 있는가?" | 릴리즈 빌드 |

각 단계는 **이전 단계 완료 기준 충족 후에만** 진입합니다.

---

## MVP 단계

### 집중 전략

**원칙**: 빠르게 만들어서 빠르게 검증. 완성도보다 핵심 경험 검증.

**집중 영역**:
1. 핵심 행동 1~2개 구현
2. 즉각적인 피드백 (소리/이펙트 없어도 시각적 상태 변화는 있어야 함)
3. 핵심 루프 1회 완주 가능 여부

**허용 사항**:
- 임시 에셋 (기본 도형, 유니티 내장 스프라이트)
- 수치 하드코딩
- 단일 씬 구성
- 임시 UI (텍스트 + 단색 버튼)

**금지 사항**:
- 완벽한 아키텍처 설계에 시간 낭비
- 없어도 되는 기능 선구현
- 그래픽 에셋 제작/구매에 과도한 투자

---

### MVP 완료 기준

```
핵심 경험 검증:
□ 플레이 테스트에서 핵심 경험 가설이 검증됨
□ 설명 없이 5분 이내에 핵심 루프를 스스로 파악한 테스터가 있음
□ 외부인 1명 이상이 자발적으로 재시작함

기술 안정성:
□ 핵심 루프 3회 이상 반복 시 크래시 없음
□ 진행 불가 상태(막힘, 루프)가 없음
□ 재시작이 항상 정상 동작함
```

---

### MVP → 알파 전환 조건

**모두 충족해야 전환 가능**:
```
□ MVP 완료 기준 전체 충족
□ 핵심 경험 가설이 검증됨 (플레이 테스트 합격 기준 통과)
□ 외부인 3명 이상이 자발적으로 재시작함
□ 치명적 버그(크래시, 진행 불가) 없음
```

**알파로 넘어가지 말아야 할 신호**:
- 테스터가 핵심 루프를 이해하지 못함
- 1회 플레이 후 재시작 의욕이 없음
- "재미있긴 한데 뭔가 부족해" → 핵심 루프 자체 재검토 필요
- 자발적 재시작률 30% 미만

**MVP 실패 시 대응**:
1. 핵심 경험 선언문 재검토 ("가장 재미있는 순간"을 다시 정의)
2. 플레이 테스트 관찰 데이터 기반으로 어느 지점에서 흥미를 잃는지 파악
3. 핵심 루프 단순화 또는 피드백 개선 후 재검증
4. 콘텐츠 추가는 하지 않음 (핵심 경험 문제)

**알파 진입 전 처리 사항**:
1. MVP 코드 리팩토링 (빠르게 짠 코드 정리)
2. 핵심 시스템 데이터 외부화 (수치를 ScriptableObject/JSON으로 분리)
3. 알파 목표 기능 목록 확정 후 동결 (이후 추가 금지)

---

## 알파 단계

### 집중 전략

**원칙**: 핵심 루프 완성 + 콘텐츠 다양성 확보 + 내부 플레이 테스트.

**집중 영역**:
1. 게임의 처음~끝 전체 루프 완성
2. 콘텐츠 다양성 추가 (적 종류, 카드 종류, 레벨 수 등)
3. 메타 시스템 기초 (세이브, 영구 진행 등)
4. 난이도 곡선 검증 (실제 수치 기반)
5. 내부 플레이 테스트 반복

**콘텐츠 추가 우선순위**:
```
1. 핵심 메카닉 변형 추가 (적 종류, 카드 종류, 타워 종류)
2. 진행 구조 완성 (스테이지, 웨이브, 층)
3. 메타 시스템 추가 (세이브, 영구 업그레이드)
4. 보조 메카닉 추가 (상점, 이벤트, 보너스)
```

---

### 알파 완료 기준

```
기능 완성:
□ 전체 게임 루프를 처음부터 끝까지 플레이 가능
□ 계획된 콘텐츠 70% 이상 구현
□ 세이브/로드 기능 동작

플레이 테스트:
□ 내부 팀 플레이 테스트 10회 이상
□ 난이도 곡선 수치 기반 검증 완료
□ 팀원이 자발적으로 "한 번 더" 하는 상태

안정성:
□ 알려진 크래시 버그 0건
□ 데이터 손실 버그 0건
```

---

### 알파 → 베타 전환 조건

**모두 충족해야 전환 가능**:
```
□ 알파 완료 기준 전체 충족
□ 전체 게임 루프 완성
□ 계획된 콘텐츠 70% 이상 구현
□ 내부 플레이 테스트 10회 이상 + 난이도 곡선 검증
□ 알려진 크래시 버그 0건
```

**베타로 넘어가지 말아야 할 신호**:
- 알파 플레이 테스트에서 새 핵심 메카닉 아이디어가 계속 나옴 → 새 아이디어는 다음 프로젝트로
- 게임의 난이도 곡선이 검증되지 않음 (너무 쉽거나 너무 어렵거나)
- 팀원이 직접 플레이하기 싫어함 (재미 문제의 신호)

## 베타 단계

### 집중 전략

**원칙**: 기능 추가 중단, 완성도와 경험 개선에 집중.

**집중 영역**:
1. 외부 플레이 테스터 모집 및 테스트
2. 버그 수정 우선
3. UI/UX 개선 (첫인상 직결)
4. 사운드 이펙트 추가 (타격감, 피드백)
5. 성능 최적화

**완성도 우선순위**:

| 순위 | 작업 | 이유 |
|------|------|------|
| 1 | 버그 수정 | 플레이 불가 상태 방지 |
| 2 | UI/UX 개선 | 첫인상에 직접 영향 |
| 3 | 사운드 이펙트 | 타격감, 즉각 피드백 |
| 4 | 비주얼 이펙트 | 주요 액션 강조 |
| 5 | 배경음악 | 몰입감 |
| 6 | 그래픽 개선 | 가장 나중 순서 |

---

### 외부 플레이 테스트 운영

**테스터 선정**:
- 타겟 유저와 유사한 사람 (게이머)
- 팀과 무관한 사람 (편향 최소화)
- 10명 이상 권장

**테스트 진행**:
```
1. 설명 없이 플레이하도록 요청
2. 화면 녹화 (허가 받고)
3. 생각 소리 내어 말하기 요청
4. 막히는 지점 즉시 메모 (개입하지 않음)
5. 끝난 후 설문/인터뷰
```

**수집 데이터**:
- 막히는 지점 (온보딩 문제)
- 재미있었던 순간 (강점 파악)
- 불편했던 점 (UX 개선 우선순위)
- 클리어까지 걸린 시간 (난이도 검증)
- 재플레이 의향 (1~5점)

**성능 최적화 목표**:
```
□ PC: 60fps (중간 사양)
□ 모바일: 30fps (3년 이상 된 기기)
□ 모바일 메모리: 300MB 이하
□ 로딩 시간: 5초 이내
```

---

### 베타 완료 기준

```
외부 테스트:
□ 외부 플레이 테스터 10명 이상 처음부터 끝까지 플레이 완료
□ 재플레이 의향 평균 3.5/5 이상

안정성:
□ 크리티컬 버그(크래시, 데이터 손실, 진행 불가) 0건
□ 마이너 버그 10건 이하

성능:
□ 목표 플랫폼 기기에서 성능 목표 달성
□ 메모리 목표 달성

완성도:
□ 핵심 UI 완성 (임시 UI 0건)
□ 핵심 액션 사운드 이펙트 완성
□ 온보딩/튜토리얼 완성 (설명 없이 5분 파악 기준 재충족)
```

---

### 베타 → 출시 준비 전환 조건

**모두 충족해야 전환 가능**:
```
□ 베타 완료 기준 전체 충족
□ 외부 플레이 테스터 10명 이상 완주
□ 크리티컬 버그 0건
□ 목표 플랫폼 성능 달성
□ 스토어 정책 위반 사항 없음 사전 확인
```

## 출시 준비 단계

### 플랫폼별 체크리스트

#### PC (Steam)

```
기술:
□ Windows 빌드 (64bit) 정상 동작
□ 해상도 설정 옵션
□ 컨트롤러 지원 (해당 시)

스토어:
□ Steam 개발자 계정 ($100)
□ 게임 페이지 (스크린샷 최소 5장, 짧은 설명, 상세 설명)
□ 트레일러 영상 (30초~2분)
□ 시스템 요구사항 명시
□ 태그 설정

출시:
□ 위시리스트 캠페인 (출시 2주 전)
□ 얼리 액세스 vs 정식 출시 결정
```

#### 모바일 (iOS/Android)

```
기술:
□ 해상도 대응 (소형~태블릿)
□ 터치 컨트롤 최적화
□ 오프라인 플레이 지원 여부 결정

Android:
□ Google Play Console 계정 ($25)
□ 앱 서명 설정
□ 개인정보 처리방침 (필수)
□ 연령 등급 설문 완료

iOS:
□ Apple Developer Program ($99/년)
□ 앱 심사 가이드라인 사전 검토
□ 개인정보 처리방침 (필수)
□ 연령 등급 설정 완료
```

### 소프트 런치 전략

```
단계 1: 클로즈드 베타 (초대 코드)
  - Steam Playtest 기능 활용
  - 모바일: TestFlight(iOS), 내부 테스트(Android)

단계 2: 소프트 런치 (일부 지역)
  - 경쟁이 낮은 지역 우선 선택
  - 크래시 데이터, D1/D7 리텐션 수집
  - 1~2주 데이터 수집 후 전체 출시 결정

단계 3: 글로벌 출시
  - 소프트 런치 데이터 기반 밸런스 조정 후 출시
```

## 출시 후 업데이트 전략

### 모니터링 지표

| 지표 | 목표 기준 | 낮을 경우 대응 |
|------|----------|--------------|
| D1 리텐션 | 30% 이상 | 온보딩 재검토 |
| D7 리텐션 | 10% 이상 | 메타 진행 속도 재검토 |
| 세션당 플레이 시간 | 계획 세션의 100% | 난이도 곡선 재검토 |
| 크래시율 | 0.5% 미만 | 즉시 핫픽스 |
| 스토어 평점 | 4.0 이상 평균 | 낮은 리뷰 원인 분석 |

### 업데이트 우선순위

```
1순위 (즉시): 크래시, 진행 불가 버그
2순위 (2주): 리텐션 저하 원인 수정
3순위 (1개월): 플레이어 요청 상위 기능
4순위 (장기): 신규 콘텐츠 추가
```

### 커뮤니티 피드백 원칙

- 버그 리포트: 24시간 이내 응대
- 부정적 리뷰: 공손하게 대응, 수정 예정 안내
- 수정 예정 항목: 공개 로드맵으로 공유 (약속한 것만 공개)
