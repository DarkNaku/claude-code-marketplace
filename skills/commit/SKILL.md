# commit — 컨텍스트 보존 커밋

## 목적

단순한 변경 요약을 넘어 **AI가 이 커밋만 읽어도 작업 배경과 의도를 복원**할 수 있는
구조화된 커밋 메시지를 생성합니다.

---

## 커밋 워크플로우

### 1. 변경 사항 수집

```bash
git status --short
git diff HEAD
git diff --staged
```

스테이징된 파일이 없으면 변경된 전체 파일을 스테이징 대상으로 파악합니다.

### 2. 컨텍스트 파일 읽기

```
Glob: work-log.md        → 있으면 Read (최근 완료 작업 파악)
Glob: designer-log.md    → 있으면 Read 마지막 항목 (기획 맥락)
Glob: developer-log.md   → 있으면 Read 마지막 항목 (구현 맥락)
```

### 3. 커밋 타입 결정

변경된 파일 경로와 내용을 분석하여 아래 타입 중 하나를 선택합니다.

| 타입 | 사용 조건 |
|------|-----------|
| `feat` | 새 기능, 새 스킬, 새 에이전트 추가 |
| `fix` | 버그 수정, 잘못된 내용 수정 |
| `refactor` | 동작 변경 없이 구조·내용 개선 |
| `docs` | README, CLAUDE.md 등 문서만 변경 |
| `test` | 테스트 코드만 변경 |
| `chore` | 설정 파일, 버전, 패키지 변경 |
| `skill` | 스킬 파일만 변경 (SKILL.md, references/) |
| `agent` | 에이전트 파일만 변경 |

변경이 여러 타입에 걸치면 **가장 핵심적인 변경** 기준으로 하나를 선택합니다.

### 4. 커밋 메시지 작성

아래 **커밋 메시지 포맷**에 따라 작성합니다.

### 5. 파일 스테이징 및 커밋

```bash
git add [변경된 파일들]
git commit -m "$(cat <<'EOF'
[작성된 커밋 메시지]
EOF
)"
```

## 커밋 메시지 포맷

```
[타입]: [한 줄 요약] (50자 이하 권장)

## 배경

[왜 이 작업을 했는지 — 요구사항, 해결하려는 문제, 이전 상태의 한계]

## 변경 내용

[파일/모듈/스킬명]: [무엇을 어떻게 변경했는지]
[파일/모듈/스킬명]: [무엇을 어떻게 변경했는지]
...

## 아키텍처 결정

[설계 판단이 필요했던 사항과 선택 이유 — 없으면 섹션 생략]

## 다음 작업

[이 커밋 이후 이어서 해야 할 작업 — 없으면 섹션 생략]
```

### 작성 규칙

- **한 줄 요약**: `[타입]: 동사+목적어` 형식, 과거형 금지 (예: `feat: setup-validator 스킬 추가`)
- **배경**: "무엇을 했는지"가 아닌 "왜 했는지" 중심으로 서술
- **변경 내용**: 파일별·모듈별로 구체적으로 나열, 추상적 표현 금지
- **아키텍처 결정**: 대안이 있었는데 이 방식을 택한 이유가 있을 때만 작성
- **다음 작업**: work-log나 에이전트 로그의 `다음 단계` 항목 참고

### 예시

```
skill: setup-validator에 교정 가이드 반복 프로세스 추가

## 배경

setup-validator가 ❌/⚠️ 항목을 발견해도 사용자가 "어떻게 고쳐야 하는지"를
알 수 없었음. 검증 결과만 알려주고 끝나는 구조라 재작업이 반복됨.
사용자가 수정 완료 후 다시 검증하는 루프가 필요했음.

## 변경 내용

skills/setup-validator/SKILL.md: 회차(N) 자동 결정 로직 추가, 결과 파일을
  setup-result-N.md로 변경, ❌/⚠️ 시 setup-guide 교정 모드 호출 절차 추가
skills/setup-guide/SKILL.md: 교정 모드 섹션 추가 (현재 상태·올바른 상태·
  차이점·수정 방법 포함한 setup-fix-N.md 출력 구조 정의)
agents/developer.md: 완료 보고 후 재검증 반복 흐름 명시

## 아키텍처 결정

교정 가이드를 setup-validator 내부에서 직접 작성하는 대신 setup-guide의
교정 모드로 위임. 가이드 작성 패턴(ide-tasks.md 참조)을 재사용할 수 있고
두 스킬의 책임을 분리하기 위함.

## 다음 작업

commit 스킬 추가 (user-invocable 커밋 커맨드)
```

## 주의사항

- **스테이징되지 않은 파일**: `git status`로 확인 후 관련 파일 전체를 스테이징
- **삭제·이동 파일 포함**: `git add -A` 대신 파일별 명시적 스테이징 권장
- **민감한 파일 제외**: `.env`, 인증 정보 등이 포함된 경우 커밋 전 사용자 확인
- **빈 커밋 방지**: 변경 사항이 없으면 커밋하지 않고 사용자에게 알림
