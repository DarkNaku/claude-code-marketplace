# pm-planner — 게임 기능 계획서 작성

## 목적

게임 개발 요청을 받아 **즉시 실행 가능한 계획서**를 작성합니다.

계획서는 **WHAT(무엇을)과 HOW(어떻게)**를 정의하는 실행 문서입니다.
- ✅ 포함: 기능 명세, 버그 원인, 해결 전략, 상세 작업 체크리스트, 성공 지표
- ❌ 제외: 비즈니스 근거(WHY), MVP 단계 구분, 수익화 분석 — 이 항목들이 필요한 경우 PRD 문서를 별도로 작성

---

## 입력 형태

다음 중 하나를 입력받습니다:

- **기능 요청**: 새로운 기능을 추가하고 싶다는 설명
- **버그 보고**: 발생한 버그 현상 + (선택) 원인 추정
- **UI/UX 개선 요청**: 화면 이미지나 텍스트로 묘사된 개선 목표
- **세 가지 조합 가능**: 예) "UI도 고치고 관련 버그도 잡아줘"

입력이 부족하면 다음을 질문합니다:
1. 요청 유형 — 신기능 / 버그 수정 / UI 개선
2. 영향받는 파일 또는 씬/컴포넌트명 (알고 있다면)
3. 기술 스택 (언어, 프레임워크, 라이브러리)

---

## 계획서 유형 판별

입력 내용을 분석하여 세 가지 유형 중 하나를 자동으로 선택합니다.

| 유형 | 판별 신호 | 섹션 2 제목 |
|------|----------|------------|
| 신기능 개발 | "추가", "구현", "만들어", "시스템", "기능" | 핵심 기능 |
| 버그 수정 | "오류", "버그", "안 됨", "이상함", "수정", "고쳐" | 버그 원인 분석 |
| UI/UX 개선 | "이미지", "화면", "디자인", "레이아웃", "UI", "UX" | 핵심 디자인 요소 |

모호한 경우 가장 유사한 유형을 선택하되, 섹션 제목을 맥락에 맞게 조정합니다.

---

## 계획서 작성 워크플로우

### 1단계: 요청 분석

- 유형 판별 (신기능 / 버그 수정 / UI 개선)
- 파일명, 씬, 클래스, 메서드 등 영향 범위 식별
- 작업 단위(Task)로 분해 가능한지 확인

### 2단계: 계획서 작성

유형에 따라 아래 템플릿 중 하나를 선택하여 작성합니다.

### 3단계: 저장

`[게임-슬러그]-[기능-슬러그].plan.md` 이름으로 저장합니다.
- 예: `link-dots-level-system.plan.md`
- 파일명에 사용할 슬러그를 사용자에게 확인하거나 요청 내용에서 자동 추출합니다.

## 작성 규칙

### 공통

1. **섹션 순서 고정**: 개요 → 분석/기능 → 기술 전략 → 작업 목록 → 성공 지표 → 상태/PDCA
2. **체크리스트 우선**: 섹션 2와 4의 항목은 반드시 `- [ ]` 형식 사용
3. **파일 경로 명시**: 섹션 4에서 수정 대상 파일을 `파일명.cs` 또는 `` `Assets/Scripts/File.cs` `` 형태로 명확히 지정
4. **성공 지표는 검증 가능하게**: "잘 된다" 대신 "클릭 시 0.1초 이내 반응", "0%로 표시됨"처럼 관찰 가능한 조건 기술

### 섹션별 세부 규칙

**섹션 2 — 유형별 포인트**
- 신기능: 기능을 그룹으로 묶어 계층적으로 작성. 각 그룹 하위에 구체적 세부 항목 들여쓰기
- 버그 수정: 문제 코드 스니펫을 포함하면 가독성이 높아짐. 원인이 여러 개면 모두 나열
- UI 개선: 레퍼런스 이미지가 있으면 "이미지 분석 반영"을 섹션 제목에 명시. 컴포넌트 단위로 분리

**섹션 3 — 기술 스택**
- 신기능: 사용하는 언어, 프레임워크, 라이브러리, 데이터 형식, 패턴 명시
- 버그 수정: "해결 전략" 섹션으로 대체 (기술 스택 나열 불필요)
- UI 개선: 사용하는 렌더링 API, 매니저 클래스, 레이아웃 전략 명시

**섹션 4 — 작업 목록**
- 파일명 + 메서드명 + 변경 내용의 3요소를 가능한 한 모두 포함
- 단순 나열보다 파일별로 묶어서 작성하면 실행하기 편함
- 마지막 항목은 검증/테스트 항목으로 마무리

## 모범 사례

1. **유형 자동 판별**: 입력 키워드로 유형을 즉시 결정하고, 불명확할 때만 질문
2. **파일 단위 작업**: 섹션 4는 파일명 기준으로 묶어서 개발자가 파일을 열지 않아도 파악 가능하게 작성
3. **성공 지표 수치화**: "~% 이상 일치", "0으로 초기화", "즉시 사라짐"처럼 관찰·측정 가능한 표현 사용
4. **코드 스니펫 활용**: 버그 수정 시 문제 코드를 인용하면 원인 이해가 빠름 — 코드가 없을 경우 생략
6. **범위 집중**: 한 계획서 = 한 목적. 신기능+버그가 섞이면 계획서를 분리하거나 명확히 구분
7. **체크리스트 완전성**: 섹션 4의 모든 항목이 완료되면 성공 지표가 자연스럽게 달성되도록 설계
