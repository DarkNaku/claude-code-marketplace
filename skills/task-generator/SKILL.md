# task-generator — Feature 단위 Task 분해

## 목적

PRD의 Feature를 **하나씩 순서대로** 구현 가능한 원자적 Task로 분해합니다.

- PRD(WHAT/WHY) → Task(HOW): Task는 구현 방법을 정의합니다
- Feature 1개 처리 완료 확인 후 다음 Feature로 진행합니다
- 각 Task는 1~4시간 이내에 완료 가능한 단위로 분해합니다

---

## 입력 형태

`game-prd-writer` 스킬이 작성한 PRD 문서를 입력받습니다.

PRD 없이 Feature 설명만 있어도 동작합니다:
- Feature 이름 + 설명
- 수용 기준 (있는 경우)

입력이 부족하면 다음을 질문합니다:
1. 이 Feature가 속한 PRD 단계 (MVP / 알파 / 베타)
2. Feature의 플레이어 시나리오 ("플레이어는 ~할 수 있다")
3. 수용 기준 (없으면 함께 도출)
4. 사용 중인 기술 스택 (Unity, 추가 라이브러리 등)

---

## 워크플로우

### 0단계: PRD Feature 목록 파악

PRD 전체를 읽고 모든 Feature를 단계별로 정리하여 먼저 제시합니다.

```
PRD Feature 전체 목록:

[MVP 필수]
  F01 · 캐릭터 이동
  F02 · 자동 공격
  F03 · 적 스폰
  ...

[알파]
  F08 · 세이브/로드
  F09 · 적 다양화
  ...

[베타]
  F12 · 사운드 이펙트
  ...

---
첫 번째로 처리할 Feature: F01 · 캐릭터 이동
다른 Feature부터 시작하려면 알려주세요.
```

### 1단계: Feature 컨텍스트 확인

처리할 Feature의 PRD 내용을 그대로 제시합니다:
- Feature 설명
- 플레이어 시나리오
- 수용 기준 (Acceptance Criteria)

수용 기준이 불명확하면 명확화 질문을 합니다.

### 2단계: Task 분해

Feature를 원자적 Task로 분해합니다.

**분해 원칙**:
- 각 Task는 **단일 책임** — 한 가지 일만 합니다
- 크기 L(4h+)은 더 쪼갭니다
- 수용 기준 1개당 최소 Task 1개 이상 대응
- 구현 → UI → 통합 → 검증 순서로 의존성 구성

**Task 카테고리**:

| 카테고리 | 정의 | 예시 |
|---------|------|------|
| **데이터** | 클래스 정의, 데이터 구조, ScriptableObject | PlayerData 클래스 정의 |
| **로직** | 핵심 알고리즘, 게임플레이 규칙 구현 | 이동 방향 계산, 충돌 처리 |
| **UI** | 화면 표시, 버튼, 레이아웃 | HP 바 표시, 레벨업 선택 패널 |
| **통합** | 시스템 간 연결, 이벤트 연결 | HP 변경 이벤트 → UI 갱신 연결 |
| **검증** | 동작 확인, 엣지 케이스 처리 | HP 0 이하 처리, 재시작 후 초기화 |

**Task 크기**:

| 크기 | 예상 시간 | 기준 |
|------|---------|------|
| **S** | ~1시간 | 단일 클래스/컴포넌트, 단순 로직 |
| **M** | 2~4시간 | 2~3개 클래스 연동, 중간 복잡도 로직 |
| **L** | 4시간+ | 더 쪼개야 함 — 반드시 분해 |

### 3단계: Task 상세 및 완료 조건 작성

각 Task에 대해:
- **할 일**: 구체적으로 무엇을 구현/작성/설정하는지
- **완료 조건**: "~가 되면 이 Task는 완료" — 검증 가능하게 작성
  - PRD의 수용 기준을 Task 완료 조건으로 분해
  - "잘 동작한다" (❌) → "입력 후 0.1초 이내에 캐릭터가 이동을 시작한다" (✅)

### 4단계: 완료 확인 및 다음 Feature

Task 목록 제시 후 대기합니다.

> 사용자가 "완료" 또는 "다음"이라고 하면 → 다음 Feature 처리 시작
> 사용자가 특정 Task에 질문하면 → 해당 Task 상세 설명
> 사용자가 Feature를 건너뛰려 하면 → 건너뛰고 다음 Feature로 이동

## 출력 형식

### Feature 처리 시작 시

```
---
## Feature F[번호]: [기능명]  |  [MVP 필수 / 알파 / 베타]

**설명**: [PRD 기능 설명]

**플레이어 시나리오**:
[플레이어 시나리오]

**수용 기준 (PRD)**:
- [ ] [AC1]
- [ ] [AC2]
- [ ] [AC3]

---
```

### Task 목록 테이블

```
### Task 목록

| ID  | Task 이름         | 카테고리 | 크기 | 선행 작업 |
|-----|-------------------|---------|------|---------|
| T01 | [Task 이름]       | 데이터   | S    | —       |
| T02 | [Task 이름]       | 로직     | M    | T01     |
| T03 | [Task 이름]       | UI      | S    | T01     |
| T04 | [Task 이름]       | 통합     | S    | T02, T03|
| T05 | [Task 이름]       | 검증     | S    | T04     |
```

### Task 상세

```
### Task 상세

**T01 — [Task 이름]** `데이터` `S`
- 할 일: [구체적으로 무엇을 구현하는지]
- 완료 조건:
  - [ ] [검증 가능한 조건 1]
  - [ ] [검증 가능한 조건 2]

**T02 — [Task 이름]** `로직` `M`
- 할 일: [구체적으로 무엇을 구현하는지]
- 완료 조건:
  - [ ] [검증 가능한 조건 1]
  - [ ] [검증 가능한 조건 2]
  - [ ] [엣지 케이스 조건]

(이하 반복)
```

### 권장 실행 순서

```
### 권장 실행 순서

T01 → T02 → T03 (병렬 가능) → T04 → T05

병렬 가능:
- T02와 T03은 T01 완료 후 동시에 작업 가능
```

### Feature 완료 안내

```
---
> **F[번호] 작업이 모두 완료되면 "완료"라고 알려주세요.**
>
> 다음 Feature: **F[다음번호] · [다음 기능명]** ([단계])
---
```

## PRD 없이 Feature 설명만 있을 때

수용 기준이 없으면 Feature 설명을 바탕으로 수용 기준을 먼저 도출하고 확인을 받습니다:

```
수용 기준이 명시되어 있지 않습니다. 아래 기준으로 진행할까요?

**도출된 수용 기준 (확인 필요)**:
- [ ] [도출된 AC1]
- [ ] [도출된 AC2]

수정이 필요하면 알려주세요. 확인되면 Task 분해를 시작합니다.
```

## 모범 사례

1. **하나씩 처리**: Feature 목록을 먼저 보여주되 처리는 1개씩, 완료 확인 후 다음으로 이동
2. **수용 기준 추적**: PRD의 각 수용 기준이 최소 1개의 Task 완료 조건에 반드시 대응
3. **크기 제한 준수**: L 크기 Task가 나오면 즉시 분해 — "이것도 더 쪼갤 수 있습니다"
4. **완료 조건은 구체적으로**: "동작한다" 대신 "어떤 입력에 어떤 결과가 나온다"
5. **검증 Task 마지막에**: 각 Feature의 마지막 Task는 반드시 수용 기준 전체 검증
6. **의존성 명시**: 병렬 작업 가능한 Task는 명시적으로 표시하여 작업 효율 높이기
7. **PRD 범위 존중**: Task를 추가하다가 PRD에 없는 기능이 포함되면 반드시 사용자에게 확인
