# Task 분해 패턴

기능 유형별 일반적인 Task 구성, 의존성 패턴, 분해 안티패턴 가이드.

---

## 기능 유형별 Task 패턴

게임에서 반복적으로 등장하는 기능 유형별 전형적인 Task 구성입니다.
실제 Task 이름과 완료 조건은 PRD의 수용 기준에 맞게 조정합니다.

---

### 캐릭터 이동 시스템

**수용 기준 예시**: 입력 방향으로 캐릭터가 이동하고, 입력 없으면 정지

```
T01 · 입력 처리 클래스 정의         [데이터, S]
T02 · 이동 방향 계산 로직           [로직, S]  ← T01
T03 · 물리/트랜스폼 이동 적용       [로직, M]  ← T02
T04 · 이동 속도 파라미터 외부화     [데이터, S] ← T03
T05 · 경계/충돌 예외 처리           [검증, S]  ← T03
T06 · 이동 동작 통합 검증           [검증, S]  ← T04, T05

의존성 그래프:
T01 → T02 → T03 → T04 ─┐
                  └─── T05 ─┤
                            ▼
                           T06
```

### HP / 피격 시스템

**수용 기준 예시**: 피격 시 HP 감소, 0이 되면 사망 이벤트 발생

```
T01 · HP 데이터 구조 정의           [데이터, S]
T02 · 피해 적용 로직                [로직, S]  ← T01
T03 · 무적 시간(Invincibility) 로직 [로직, S]  ← T02
T04 · HP 0 사망 이벤트 발생         [로직, S]  ← T02
T05 · HP 바 UI 컴포넌트             [UI, M]    ← T01
T06 · HP 변경 → UI 갱신 연결        [통합, S]  ← T02, T05
T07 · 피격/사망 엣지 케이스 검증    [검증, S]  ← T03, T04, T06

병렬 가능: T03, T04, T05 모두 T02 완료 후 동시에 진행 가능
```

### 레벨업 / 경험치 시스템

**수용 기준 예시**: 경험치 목표 달성 시 레벨업, 선택지 3개 제시

```
T01 · 경험치/레벨 데이터 구조 정의  [데이터, S]
T02 · 경험치 획득 로직              [로직, S]  ← T01
T03 · 레벨업 임계값 계산 로직       [로직, S]  ← T01
T04 · 레벨업 이벤트 발생            [로직, S]  ← T02, T03
T05 · 선택지 데이터 정의 (SO 등)    [데이터, S]
T06 · 선택지 3개 무작위 추출 로직   [로직, S]  ← T05
T07 · 레벨업 선택 패널 UI           [UI, M]    ← T05
T08 · 레벨업 발생 → 패널 표시 연결  [통합, S]  ← T04, T07
T09 · 선택 완료 → 능력 적용 연결    [통합, S]  ← T06, T08
T10 · 레벨업 흐름 전체 검증         [검증, S]  ← T09

병렬 가능: T05~T07은 T04와 무관하게 진행 가능
```

### 적 스폰 시스템

**수용 기준 예시**: 시간에 따라 적이 화면 외곽에서 등장하고 수가 점진적으로 증가

```
T01 · 스폰 데이터 구조 정의         [데이터, S]
     (스폰 위치, 간격, 적 종류 등)
T02 · 화면 외곽 스폰 위치 계산 로직 [로직, S]  ← T01
T03 · 스폰 타이머 / 간격 로직       [로직, S]  ← T01
T04 · 적 인스턴스 생성 (오브젝트 풀 또는 Instantiate) [로직, M] ← T01, T02
T05 · 시간에 따른 스폰 빈도 증가    [로직, S]  ← T03
T06 · 스폰 데이터 파라미터 외부화   [데이터, S] ← T04
T07 · 스폰 시작/정지 제어           [통합, S]  ← T03, T04
T08 · 스폰 동작 검증                [검증, S]  ← T05, T07

병렬 가능: T02, T03은 T01 완료 후 동시에 진행 가능
```

### 자동 공격 시스템

**수용 기준 예시**: 가장 가까운 적을 일정 주기로 자동 공격, 범위 내 적 없으면 공격 없음

```
T01 · 공격 데이터 구조 정의         [데이터, S]
     (사거리, 쿨다운, 데미지)
T02 · 범위 내 적 탐색 로직          [로직, S]  ← T01
T03 · 가장 가까운 적 선택 로직      [로직, S]  ← T02
T04 · 공격 쿨다운 타이머 로직       [로직, S]  ← T01
T05 · 공격 발동 → 피해 시스템 호출  [통합, S]  ← T03, T04
T06 · 공격 시각/사운드 피드백       [UI, S]    ← T05
T07 · 자동 공격 동작 검증           [검증, S]  ← T05, T06
     (범위 내/외 적, 쿨다운 정확도)
```

### 카드 드로우 / 플레이 시스템

**수용 기준 예시**: 턴 시작 시 드로우, 에너지 소비하여 카드 플레이, 버린 덱 재활용

```
T01 · 카드 데이터 구조 정의         [데이터, S]
T02 · 덱/손패/버린덱 컨테이너 정의  [데이터, S]
T03 · 드로우 로직                   [로직, S]  ← T01, T02
     (드로우 덱 소진 시 버린덱 재활용 포함)
T04 · 에너지 시스템                 [로직, S]
     (턴당 충전, 카드 플레이 시 소모)
T05 · 카드 플레이 로직              [로직, M]  ← T01, T04
     (에너지 검증 → 효과 적용 → 버린덱 이동)
T06 · 손패 카드 UI 표시             [UI, M]    ← T01, T02
T07 · 에너지 UI 표시                [UI, S]    ← T04
T08 · 카드 드래그 플레이 인터랙션   [UI, M]    ← T05, T06
T09 · 드로우/플레이/버리기 흐름 검증 [검증, M] ← T03~T08
     (덱 소진, 에너지 부족 등 엣지 케이스)

병렬 가능:
- T03, T04는 T01,T02 완료 후 동시 진행 가능
- T06, T07은 각 선행 완료 후 동시 진행 가능
```

### 3매치 퍼즐 보드

**수용 기준 예시**: 스와이프로 타일 교환, 3개 이상 매칭 시 제거 및 연쇄 처리

```
T01 · 보드 데이터 구조 정의         [데이터, S]
     (그리드 크기, 타일 타입, 좌표 체계)
T02 · 보드 초기화 및 타일 배치 로직 [로직, M]  ← T01
T03 · 스와이프 입력 처리            [로직, S]
T04 · 타일 교환 유효성 검사 로직    [로직, M]  ← T01, T03
     (매칭 성립 시에만 교환)
T05 · 3개 이상 매칭 탐색 로직       [로직, M]  ← T01
T06 · 매칭 타일 제거 로직           [로직, S]  ← T05
T07 · 타일 낙하(Gravity) 로직       [로직, M]  ← T06
T08 · 낙하 후 재매칭(연쇄) 로직     [로직, M]  ← T07
T09 · 보드 UI 렌더링                [UI, M]    ← T01, T02
T10 · 스와이프 애니메이션/이펙트    [UI, M]    ← T04, T09
T11 · 제거/낙하 애니메이션          [UI, M]    ← T06, T07, T09
T12 · 이동 수 카운터                [로직, S]
T13 · 보드 전체 동작 검증           [검증, M]  ← T04~T12

주의: T09는 T01,T02 완료 후 T05~T08과 병렬로 시작 가능
```

### 오브젝트 병합 시스템

**수용 기준 예시**: 같은 오브젝트 2개 겹치면 상위 오브젝트 1개로 합쳐짐

```
T01 · 오브젝트 타입 계층 데이터 정의 [데이터, S]
     (A+A=B, B+B=C... 병합 트리)
T02 · 보드 그리드 데이터 구조        [데이터, S]
T03 · 드래그 앤 드롭 입력 처리       [로직, M]  ← T02
T04 · 병합 가능 여부 판별 로직       [로직, S]  ← T01, T02
T05 · 병합 실행 로직                 [로직, S]  ← T01, T04
     (원본 2개 제거, 결과 1개 생성)
T06 · 보드 칸 관리 로직              [로직, S]  ← T02
     (빈 칸 확인, 배치 가능 여부)
T07 · 오브젝트 UI 표시               [UI, M]    ← T01
T08 · 드롭 시 병합/원위치 피드백     [UI, S]    ← T03, T05
T09 · 신규 오브젝트 등장 강조 이펙트 [UI, S]    ← T05
T10 · 병합 동작 검증                 [검증, S]  ← T05~T09
     (병합 불가 시 원위치, 보드 꽉 참 처리)
```

### 타워 배치 시스템

**수용 기준 예시**: 배치 가능 영역에 타워 클릭/드래그로 배치, 골드 소비

```
T01 · 타워 데이터 정의               [데이터, S]
     (타입, 비용, 사거리, 공격력)
T02 · 배치 가능 영역 맵 데이터       [데이터, S]
T03 · 타워 선택 UI                   [UI, S]    ← T01
T04 · 배치 위치 유효성 검사 로직     [로직, S]  ← T02
     (적 경로 위 배치 불가)
T05 · 타워 배치 실행 로직            [로직, S]  ← T01, T04
T06 · 골드 차감 로직                 [로직, S]  ← T01, T05
T07 · 배치 프리뷰(고스트 이미지)     [UI, M]    ← T03, T04
T08 · 배치된 타워 사거리 표시        [UI, S]    ← T07
T09 · 골드 부족 시 피드백            [UI, S]    ← T06
T10 · 배치 흐름 검증                 [검증, S]  ← T05~T09

병렬 가능: T03, T04는 각각 독립적으로 T01,T02 완료 후 시작 가능
```

### 세이브 / 로드

**수용 기준 예시**: 게임 종료 후 재실행 시 이전 진행 상태 복원

```
T01 · 저장할 데이터 구조 정의        [데이터, S]
     (무엇을 저장할지 목록 확정)
T02 · 직렬화/역직렬화 구현           [로직, M]  ← T01
T03 · 저장 경로/파일 관리            [로직, S]  ← T01
T04 · 세이브 트리거 시점 구현        [통합, S]  ← T02, T03
     (씬 전환, 앱 일시정지, 주기적 자동저장)
T05 · 로드 트리거 시점 구현          [통합, S]  ← T02, T03
     (앱 시작 시)
T06 · 저장 데이터 없을 때 기본값 처리 [검증, S] ← T05
T07 · 세이브/로드 검증               [검증, M]  ← T04, T05, T06
     (저장-재시작-로드 전체 흐름)

주의: 저장할 데이터 목록(T01)이 다른 Feature들의 구현 이후에 확정되는 경우가 많음
→ 세이브/로드는 다른 Feature들이 어느 정도 구현된 후 진행 권장
```

## 의존성 그래프 구성 방법

### 일반적인 레이어 순서

```
데이터 구조 정의
    ↓
핵심 로직 구현 (데이터 기반)
    ↓
UI 컴포넌트 구현 (데이터/로직 기반) ──┐ (병렬 가능)
이벤트/시스템 연결 (로직 기반)        ──┘
    ↓
통합 연결 (로직 ↔ UI)
    ↓
검증 (전체 흐름 + 엣지 케이스)
```

### 병렬화 기회

같은 레이어에 있는 Task들은 대부분 병렬 진행 가능합니다:
- 데이터 구조가 2개 이상 있을 때: 서로 독립적이면 동시 정의 가능
- UI 컴포넌트가 여러 개: 같은 데이터에 의존하면 동시 구현 가능
- 여러 이벤트 연결: 각각 독립적이면 동시 작업 가능

## Task 분해 안티패턴

### 안티패턴 1: 너무 큰 Task (크기 L)

```
❌ T01 · 전투 시스템 구현  [로직, L]

✅ T01 · 피해 계산 로직         [로직, S]
   T02 · 피격 이벤트 발생       [로직, S]  ← T01
   T03 · 무적 시간 처리         [로직, S]  ← T02
   T04 · 사망 판정 로직         [로직, S]  ← T01
```

크기 L이 나오면 반드시 분해합니다. "어떤 단계로 나눌 수 있을까?"를 질문.

---

### 안티패턴 2: 너무 작은 Task (nano task)

```
❌ T01 · hp 변수 선언      [데이터, XS]
   T02 · hp getter 추가    [데이터, XS]
   T03 · hp setter 추가    [데이터, XS]

✅ T01 · HP 데이터 구조 정의  [데이터, S]
        (hp 필드, 최댓값, 현재값, 변경 로직 포함)
```

30분 미만으로 추정되는 Task는 하나로 묶습니다.

---

### 안티패턴 3: 완료 조건이 모호한 Task

```
❌ 완료 조건: 이동이 잘 된다

✅ 완료 조건:
   - [ ] WASD/방향키 입력 시 해당 방향으로 즉시 이동 시작
   - [ ] 입력 없을 때 정지
   - [ ] 대각선 이동 시 속도가 수평/수직과 동일 (정규화 처리)
   - [ ] 화면 경계에서 이동 불가 또는 루프 처리
```

### 안티패턴 4: PRD에 없는 기능 Task 추가

```
상황: HP 시스템 Task를 분해하다가 "회복 포션도 있으면 좋겠다"는 Task 추가

❌ T07 · 회복 포션 시스템 구현  (PRD에 없음)

올바른 처리:
→ "T07 · 회복 포션 시스템이 PRD에 포함되어 있지 않습니다.
   이 Feature에 추가할까요, 아니면 별도 Feature로 PRD에 추가할까요?"
```

PRD 범위를 벗어나는 Task 발견 시 반드시 사용자 확인.

### 안티패턴 5: 수용 기준과 연결되지 않은 Task

```
PRD 수용 기준:
- [ ] 이동 수 0이 되면 즉시 실패 처리

❌ 이 수용 기준에 대응하는 Task가 없음

올바른 처리:
→ 수용 기준 목록을 먼저 정리하고
→ 각 수용 기준이 최소 1개의 Task 완료 조건에 포함되는지 확인
→ 누락된 수용 기준이 있으면 Task 추가
```

## Feature 완료 체크리스트

각 Feature의 모든 Task 완료 후 다음을 확인합니다:

```
□ PRD의 모든 수용 기준이 Task 완료 조건에 대응됨
□ 모든 Task 완료 조건이 실제로 통과됨
□ 엣지 케이스 (최솟값, 최댓값, 빈 상태) 처리 완료
□ 다른 이미 완료된 Feature와의 통합에 문제 없음
□ PRD에서 "의도적 제외"로 명시한 기능이 포함되지 않음
```
