# Unity TDD - 테스트 주도 개발

## 핵심 철학

Kent Beck의 테스트 주도 개발(TDD)과 Tidy First 원칙을 정확하게 따르며 개발을 안내한다.

한 번에 하나의 테스트만 작성하고, 통과시키고, 구조를 개선한다. 빠른 구현보다 깨끗하고 잘 테스트된 코드를 우선시한다.

---

## TDD 방법론

- 작은 기능 증분을 정의하는 실패하는 테스트부터 작성하세요
- 동작을 설명하는 의미 있는 테스트 이름을 사용하세요
- 테스트 실패 메시지는 명확해야 합니다
- 테스트를 통과시키는 최소한의 코드만 작성하세요
- 테스트 통과 후에만 리팩토링하세요
- **한 번에 하나의 테스트만** 작성하고, 통과시킨 후 다음으로 진행하세요

### 결함 수정 절차

결함을 수정할 때는 다음 순서를 따릅니다:
1. API 레벨의 실패하는 테스트 작성 (전체 동작 검증)
2. 결함을 재현하는 가장 작은 단위 테스트 작성
3. 두 테스트를 모두 통과시키는 최소한의 코드 구현

---

## Tidy First 접근법

### 변경 유형 구분

| 유형 | 정의 | 예시 |
|------|------|------|
| **구조적 변경** | 동작 변경 없이 코드 구조 개선 | 이름 변경, 메서드 추출, 코드 이동, 클래스 분리 |
| **행동 변경** | 실제 기능 추가 또는 수정 | 새 로직 구현, 버그 수정 |

### 규칙
- 두 변경 유형을 같은 커밋에 절대 섞지 마세요
- 두 가지가 모두 필요할 때는 **구조적 변경을 먼저** 수행하세요
- 구조적 변경 전후에 테스트를 실행하여 동작이 동일함을 검증하세요

---

## 워크플로우

1. **Red**: 작은 기능 하나에 대한 실패하는 테스트 작성
2. **Green**: 테스트를 통과시키는 최소한의 코드 구현
3. 테스트 통과 확인
4. **Tidy First (선택)**: 필요한 구조적 변경 수행, 각 단계마다 테스트 실행
5. 구조적 변경을 별도 커밋으로 저장
6. **Refactor**: 테스트를 유지하며 중복 제거 및 명확성 개선
7. 행동 변경을 별도 커밋으로 저장
8. 반복

---

## 커밋 규율

다음 조건이 모두 충족될 때만 커밋합니다:
1. **모든 테스트 통과**
2. **컴파일러/린터 경고 없음**
3. **단일 논리적 작업 단위** (구조적 변경과 행동 변경 분리)
4. **커밋 메시지에 변경 유형 명시** — 구조적 변경인지 행동 변경인지 기재

소규모 빈번한 커밋을 선호합니다. 대규모 커밋은 지양합니다.

---

## 코드 품질 기준

- **중복 제거**: 중복 코드를 적극적으로 제거하세요
- **의도 표현**: 이름과 구조로 코드의 의도를 명확하게 드러내세요
- **명시적 의존성**: 의존성을 숨기지 말고 명시하세요
- **작은 메서드**: 메서드는 단일 책임에 집중하고 작게 유지하세요
- **상태와 부작용 최소화**: 가변 상태와 side effect를 최소화하세요
- **가장 단순한 해결책**: 작동할 수 있는 가장 단순한 코드를 사용하세요

---

## 리팩토링 가이드라인

- 테스트가 통과된 상태(Green)에서만 리팩토링하세요
- 확립된 리팩토링 패턴을 이름과 함께 사용하세요 (Extract Method, Rename Variable 등)
- 한 번에 하나의 리팩토링만 적용하세요
- 각 리팩토링 단계 후 테스트를 실행하세요
- 중복 제거와 명확성 개선을 우선 순위로 삼으세요
- 리팩토링 중 테스트가 깨지면 즉시 되돌리세요

## 빠른 시작

```csharp
// 1. RED - 실패하는 테스트 작성
[Test]
public void NewPlayer_HasFullHealth()
{
    var player = new Player();
    Assert.AreEqual(100, player.Health);
}
// 실행 → 컴파일 에러 (Player 클래스 없음) ← 의도된 실패

// 2. GREEN - 최소한의 구현 (가장 단순한 것부터)
public class Player
{
    public int Health => 100; // 현재 테스트를 통과하는 가장 단순한 구현
}
// 실행 → 테스트 통과 ✓ → 커밋

// 3. RED - 다음 테스트
[Test]
public void TakeDamage_ReducesHealth()
{
    var player = new Player();
    player.TakeDamage(30);
    Assert.AreEqual(70, player.Health);
}
// 실행 → 실패 (TakeDamage 메서드 없음)

// 4. GREEN - 최소한의 구현
public class Player
{
    public int Health { get; private set; } = 100;

    public void TakeDamage(int damage)
    {
        Health -= damage;
    }
}
// 실행 → 모든 테스트 통과 ✓

// 5. REFACTOR (필요 시) → 커밋
```

## 모범 사례

1. **작은 단계**: 한 번에 하나의 작은 기능만 테스트
2. **빠른 피드백**: 테스트 실행 주기를 짧게 유지 (몇 초 이내)
3. **테스트 먼저**: 구현 전에 항상 테스트 먼저 작성
4. **최소 구현**: 테스트를 통과시키는 가장 단순한 코드
5. **리팩토링 자신감**: 테스트가 있으면 두려움 없이 리팩토링
6. **분리된 커밋**: 구조적 변경과 행동 변경을 별도 커밋으로 분리
7. **경고 해소**: 커밋 전 컴파일러 경고를 모두 제거
8. **결함은 테스트로 증명**: 수정 전에 결함을 재현하는 테스트 먼저 작성
