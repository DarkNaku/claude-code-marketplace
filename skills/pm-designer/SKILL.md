# pm-designer — 게임 기능 설계서 작성

## 목적

계획서(WHAT)를 받아 **개발자가 즉시 코딩을 시작할 수 있는 설계서(HOW)**를 작성합니다.

설계서는 **HOW(어떻게 구현할지)**를 정의하는 실행 명세입니다.
- ✅ 포함: 클래스 구조, 데이터 스키마, 메서드 시그니처, 알고리즘 흐름, 좌표/크기/색상 수치, 코드 스니펫, 수정 스펙 표, 검증 계획
- ❌ 제외: WHAT(기능 목표), WHY(비즈니스 근거), 체크리스트 — 이것들은 계획서의 영역

---

## 입력 형태

다음 중 하나를 입력받습니다:

- **계획서 파일 경로**: `.plan.md` 파일의 경로 또는 내용
- **계획서 내용 직접 전달**: 계획서 텍스트를 붙여넣기
- **구현 요청**: 설계가 필요한 기능/버그/UI를 직접 설명

계획서 없이 요청이 들어오면:
1. 대상 파일 또는 씬/클래스명
2. 기술 스택 (언어, 프레임워크, 라이브러리)
3. 기존 코드 구조 (관련 파일을 Read 툴로 직접 확인)

---

## 설계서 유형 판별

계획서의 섹션 2 제목으로 유형을 자동 판별합니다.

| 계획서 섹션 2 제목 | 설계서 유형 | 핵심 설계 내용 |
|------------------|------------|--------------|
| 핵심 기능 | 신기능 설계 | 아키텍처, 데이터 스키마, 클래스/메서드 설계, 알고리즘 |
| 버그 원인 분석 | 버그 수정 설계 | 수정 로직, 코드 스니펫, 수정 스펙 표, 검증 계획 |
| 핵심 디자인 요소 | UI/UX 설계 | 컴포넌트 수치, 레이아웃 좌표, 이벤트 연동, 예외 처리 |

판별이 모호하면 계획서를 Read 툴로 직접 읽어 확인합니다.

---

## 설계서 작성 워크플로우

### 1단계: 계획서 분석

계획서가 있으면 반드시 Read 툴로 내용을 읽습니다.
- 유형 판별 (신기능 / 버그 수정 / UI 개선)
- 영향받는 파일, 클래스, 메서드 목록 추출
- 기존 코드가 있으면 Glob/Grep/Read 툴로 실제 코드 확인

### 2단계: 기존 코드 파악 (필요 시)

설계가 기존 코드에 의존하는 경우:
- Glob으로 관련 파일 탐색
- Read로 핵심 클래스/메서드 구조 확인
- Grep으로 특정 변수·함수 사용처 파악

### 3단계: 설계서 작성

유형에 따라 아래 형식 중 하나로 작성합니다.

### 4단계: 저장

계획서와 동일한 슬러그를 사용합니다.
- 계획서: `[slug].plan.md` → 설계서: `[slug].design.md`
- 저장 경로를 사용자에게 확인하거나 계획서 경로에서 유추합니다.

## 설계서 출력 형식

### 유형 A: 신기능 설계

신기능의 규모에 따라 필요한 섹션을 선택합니다. 섹션 수는 3~5개로 가변입니다.

```markdown
# 설계서: [기능명]

## 1. 시스템 아키텍처
[전체 구조, 사용 프레임워크, 씬/모듈 구성 개요]

### 1.1 [설정 또는 구조 항목]
- **[속성]**: [값 또는 설명]

## 2. 데이터 구조 설계

### 2.1 [데이터 이름] ([형식: JSON / 클래스 속성 / etc.])
[스키마 또는 클래스 속성 목록]
```[언어]
{
  "[필드]": "[타입/값]",
  ...
}
```

## 3. 클래스 및 메서드 설계

### 3.1 `[ClassName]`
- `[methodName]([params])`: [역할 설명]
- `[methodName]([params])`: [역할 설명]

## 4. 핵심 로직 및 알고리즘

### 4.1 [로직 이름]
1. [순서 1]
2. [순서 2]
3. [조건 분기 및 처리]

## 5. UI 및 시각적 요소 (해당하는 경우)
- **[요소]**: [구현 방식 및 수치]

```

---

### 유형 B: 버그 수정 설계

```markdown
# 설계서: [버그명 또는 수정 목표]

## 1. 개요
[계획서의 버그 현상을 1~2문장으로 재기술. 수정의 핵심 방향 제시]

## 2. 세부 설계 사항

### 2.1 [수정 대상 파일/컴포넌트] (`파일명.확장자`)
- **로직**: [변경할 로직의 동작 방식 설명]
- **조건**: [적용 조건 또는 분기]
- **메서드**: [호출할 메서드 또는 수정할 코드]

### 2.2 [다음 수정 대상] (`파일명.확장자`)
- **[이벤트 또는 상황]**:
    - [동작 설명 1]
    - [동작 설명 2]

## 3. 상세 수정 스펙

| 파일 | 위치 | 수정 내용 |
| :--- | :--- | :--- |
| `[파일 경로]` | `[메서드명 또는 위치]` | [구체적 수정 내용] |
| `[파일 경로]` | `[메서드명 또는 위치]` | [구체적 수정 내용] |

## 4. 기대 결과 / 검증 계획
- [수정 후 예상되는 정상 동작 1]
- [수정 후 예상되는 정상 동작 2]

```

## 작성 규칙

### 공통

1. **체크리스트 금지**: 설계서에는 `- [ ]` 형식을 사용하지 않음. 모든 항목은 확정된 명세로 서술
2. **수치 명시**: 좌표, 크기, 색상, 투명도 등은 반드시 실제 값으로 기술 (`width * 0.1`, `#00FF00`, `0.35` 등)
3. **코드 스니펫 적극 활용**: 로직이 복잡하거나 의사 코드가 이해를 돕는 경우 반드시 포함
4. **수정 스펙 표 (버그 수정 필수)**: `파일 | 위치 | 수정 내용` 3열 표로 정리

### 섹션별 세부 규칙

**신기능 — 섹션 구성 판단 기준**
- 데이터 모델이 새로 생기면: "데이터 구조 설계" 섹션 필수
- 새로운 클래스/매니저가 생기면: "클래스 및 메서드 설계" 섹션 필수
- 복잡한 상태 전환/판정 로직이 있으면: "핵심 로직 및 알고리즘" 섹션 필수
- 렌더링/시각 요소가 있으면: "UI 및 시각적 요소" 섹션 추가

**버그 수정 — 세부 설계 사항 작성 원칙**
- 파일 단위로 섹션 분리 (`### 2.1 PathManager.js`, `### 2.2 MainScene.js`)
- 이벤트 핸들러 수정 시 이벤트명과 콜백 내 변경 사항을 명확히 기술
- 수정 스펙 표는 실행 순서대로 나열

**UI 개선 — 컴포넌트 설계 작성 원칙**
- 메서드 시그니처를 섹션 제목으로 사용 (`### 1.1 UIManager.createHeader(level)`)
- 좌표는 하드코딩 값 대신 비율식 사용 권장 (`this.width * 0.1`)
- 색상은 Hex 코드 명시 (`#FF0000`, `0xFF0000`)
- Depth 값이 있으면 다른 요소와의 레이어 관계를 함께 기술

## 모범 사례

1. **계획서를 먼저 읽는다**: 설계 시작 전 반드시 Read 툴로 계획서를 확인하여 범위와 요구사항을 파악
2. **기존 코드를 확인한다**: 수정 대상 파일이 있으면 Read/Grep 툴로 현재 구조를 파악한 후 설계
3. **수치는 구체적으로**: "적절한 크기" 대신 `cellSize * 0.8`, "화면 하단" 대신 `height - 100`
4. **코드 스니펫은 의사 코드도 가능**: 완전한 코드가 아니어도 핵심 로직이 드러나면 충분
5. **버그 수정 설계의 핵심은 표**: 수정 스펙 표(`파일 | 위치 | 수정 내용`)를 보면 작업 범위가 한눈에 보여야 함
6. **예외 처리를 명시**: 엣지 케이스(빈 배열, null, 초기화 타이밍 등)를 설계 단계에서 결정
7. **슬러그 일관성**: 계획서 슬러그와 설계서 슬러그를 동일하게 유지하여 추적 가능하게 관리
