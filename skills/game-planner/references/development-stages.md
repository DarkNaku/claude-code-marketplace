# 단계별 개발 전략

MVP → 알파 → 베타 → 출시 준비 각 단계에서 집중할 것과 전환 기준 가이드.

---

## 단계 전환 기준

### MVP → 알파 전환

**전환 조건** (모두 충족):
```
□ 핵심 재미가 플레이 테스트에서 검증됨
□ 외부인 3명 이상이 자발적으로 재시작함
□ 치명적인 버그 없음 (크래시, 진행 불가)
□ 핵심 루프를 3회 이상 반복 가능
```

**MVP에서 알파로 넘어가지 말아야 할 신호**:
- 테스터가 핵심 루프를 이해하지 못함
- 1회 플레이 후 재시작 의욕이 없음
- "재미는 있는데 뭔가 부족해" → 핵심 루프 자체를 재검토 필요

**알파 시작 전 해야 할 것**:
1. MVP 코드 리팩토링 (빠르게 짠 코드 정리)
2. 폴더 구조 정리
3. 핵심 시스템 단위 테스트 작성
4. 알파 목표 기능 목록 확정 (더 이상 추가 금지)

---

### 알파 → 베타 전환

**전환 조건** (모두 충족):
```
□ 전체 게임 루프 완성 (처음부터 끝까지 플레이 가능)
□ 계획된 콘텐츠 70% 이상 구현
□ 내부 팀 플레이 테스트 10회 이상
□ 알려진 크래시 버그 0건
□ 핵심 난이도 곡선 검증됨
```

**알파에서 베타로 넘어가지 말아야 할 신호**:
- 알파 플레이 테스트에서 새로운 핵심 메카닉 아이디어가 계속 나옴
  → 새 아이디어는 다음 프로젝트로. 지금 프로젝트는 완성에 집중.
- 게임의 난이도 곡선이 검증되지 않음
- 팀원이 직접 플레이하기 싫어함 (재미 문제 신호)

---

### 베타 → 출시 준비 전환

**전환 조건** (모두 충족):
```
□ 외부 플레이 테스터 10명 이상 완주
□ 크리티컬 버그 0건, 마이너 버그 10건 이하
□ 목표 플랫폼 기기에서 60fps (모바일 30fps) 달성
□ 게임 클리어까지 예상 플레이 시간 검증
□ 스토어 정책 위반 사항 없음 확인
```

## 각 단계별 집중 전략

### MVP 단계: 속도 최우선

**원칙**: 빠르게 만들어서 빠르게 검증

**코딩 전략**:
- 하드코딩 허용 (수치, 문자열 등)
- 추상화/패턴 최소화
- 임시 에셋 사용 (Unity 기본 도형, Asset Store 무료)
- 단일 씬으로 모든 것 처리 가능

**피해야 할 것**:
- 완벽한 아키텍처 설계에 시간 낭비
- 없어도 되는 기능을 "나중에 필요할 것 같아서" 미리 구현
- 에셋 검색/구매에 과도한 시간 투자

**예시 빠른 구현 패턴**:
```csharp
// MVP: 하드코딩으로 빠르게
public class PlayerStats : MonoBehaviour
{
    public float mMoveSpeed = 5f;
    public int mMaxHp = 100;
    public float mAttackCooldown = 0.5f;
}

// 알파 리팩토링: ScriptableObject로 전환
[CreateAssetMenu]
public class PlayerStatData : ScriptableObject
{
    public float mMoveSpeed = 5f;
    public int mMaxHp = 100;
    public float mAttackCooldown = 0.5f;
}
```

---

### 알파 단계: 루프 완성 + 리팩토링

**원칙**: 핵심 루프를 완성하고, 확장 가능한 구조로 정리

**집중 영역**:
1. **시스템 확장성**: 콘텐츠(적, 무기, 카드 등)를 코드 수정 없이 추가 가능한 구조
2. **데이터 분리**: 수치를 ScriptableObject 또는 JSON으로 외부화
3. **난이도 곡선**: 실제 수치 기반의 진행 속도 검증
4. **버그 수정 우선**: 새 기능보다 기존 기능 안정화

**알파 아키텍처 패턴**:

```
데이터 분리:
Scripts/Data/         ← ScriptableObject 데이터
Scripts/Core/         ← 핵심 게임 로직 (재사용 가능)
Scripts/Systems/      ← 게임 시스템 (SpawnSystem, CombatSystem 등)
Scripts/UI/           ← UI 전용 스크립트
Scripts/Utils/        ← 유틸리티 (ObjectPool, EventBus 등)

오브젝트 풀링 도입:
- 자주 생성/삭제되는 오브젝트 (총알, 이펙트, 적)
- ObjectPool<T> 또는 Unity의 ObjectPool 사용

이벤트 시스템 도입:
- 시스템 간 직접 참조 줄이기
- UnityEvent 또는 C# Action/Event 활용
```

**콘텐츠 추가 순서**:
```
1. 핵심 메카닉 변형 추가 (적 종류, 무기 종류)
2. 진행 구조 완성 (스테이지, 웨이브, 층)
3. 메타 시스템 추가 (세이브, 영구 업그레이드)
4. 보조 메카닉 추가 (상점, 이벤트, 보너스)
```

---

### 베타 단계: 완성도 + 외부 테스트

**원칙**: 기능 추가 중단, 품질과 경험 개선에 집중

**완성도 작업 우선순위**:

| 우선순위 | 작업 | 이유 |
|---|---|---|
| 1 | 버그 수정 | 플레이 불가 상태 방지 |
| 2 | UI/UX 개선 | 첫인상에 직접 영향 |
| 3 | 사운드 이펙트 | 타격감, 피드백 |
| 4 | 비주얼 이펙트 | 주요 액션 강조 |
| 5 | 배경음악 | 몰입감 |
| 6 | 그래픽 개선 | 최후 순서 |

**외부 플레이 테스트 운영**:
```
테스터 모집:
- 타겟 유저와 유사한 사람 (게이머)
- 팀과 관련 없는 사람 (편향 최소화)
- 10명 이상 권장

진행 방법:
1. 설명 없이 플레이하도록 요청
2. 화면 녹화 (허가 받고)
3. 생각 소리 내어 말하기 요청
4. 끝난 후 설문/인터뷰

수집할 데이터:
- 막히는 지점
- 재미있었던 순간
- 불편했던 점
- 클리어까지 걸린 시간
- 재플레이 의향 (1~5점)
```

**성능 최적화 체크리스트**:
```
프로파일링:
□ Unity Profiler로 CPU/GPU 병목 식별
□ 메모리 누수 확인 (GC 스파이크)

렌더링:
□ Draw Call 최적화 (스프라이트 아틀라스)
□ LOD 설정 (3D 게임)
□ 라이팅 베이크 (정적 오브젝트)

스크립트:
□ Update에서 불필요한 GetComponent 제거
□ 오브젝트 풀 활용 확인
□ Coroutine vs Update 적절히 사용

목표:
□ PC: 60fps (중간 사양)
□ 모바일: 30fps (3년 이상 된 기기)
□ 메모리: 300MB 이하 (모바일)
```

## 출시 준비 단계

### 플랫폼별 체크리스트

#### PC (Steam)

```
기술:
□ Windows 빌드 (64bit)
□ macOS 빌드 (선택)
□ 해상도 설정 옵션
□ 키보드/마우스 + 컨트롤러 지원

스토어:
□ Steam 개발자 계정 ($100)
□ 게임 페이지 (최소 5장 스크린샷)
□ 트레일러 영상 (30초~2분)
□ 게임 설명 (영어 필수, 한국어 선택)
□ 시스템 요구사항
□ 태그 설정 (장르, 기능)

출시:
□ 위시리스트 캠페인 (2주 전)
□ 얼리 액세스 vs 정식 출시 결정
```

#### 모바일 (iOS/Android)

```
기술:
□ 해상도 대응 (iPhone SE ~ iPad Pro)
□ 터치 컨트롤 최적화
□ 배터리/발열 최적화
□ 오프라인 플레이 가능 여부

Android (Google Play):
□ Google Play Console 계정 ($25)
□ 앱 서명 설정
□ 대상 SDK 버전 (최신 Android 요구사항)
□ 개인정보 처리방침 (필수)
□ 연령 등급 설문

iOS (App Store):
□ Apple Developer Program ($99/년)
□ Xcode 빌드 설정
□ 앱 심사 가이드라인 검토
□ 개인정보 처리방침 (필수)
□ 연령 등급 설정
```

### 소프트 런치 전략

출시 직후 모든 사용자에게 공개하기 전 소규모로 테스트.

```
단계:
1. 클로즈드 베타 (초대 코드)
   - Steam: Playtest 기능
   - 모바일: TestFlight(iOS), 내부 테스트(Android)

2. 소프트 런치 (일부 국가)
   - 경쟁이 덜한 국가 먼저 (예: 동남아 국가)
   - 크래시 데이터, 리텐션 지표 수집
   - 1~2주 후 정식 출시 결정

3. 글로벌 출시
   - 소프트 런치 데이터 기반 밸런스 조정 후 출시
```

### 분석 도구 연동

```csharp
// Firebase Analytics 예시
using Firebase.Analytics;

public class GameAnalytics : MonoBehaviour
{
    // 핵심 이벤트 트래킹
    public static void TrackLevelStart(int level)
    {
        FirebaseAnalytics.LogEvent("level_start",
            new Parameter("level_number", level));
    }

    public static void TrackLevelComplete(int level, float duration)
    {
        FirebaseAnalytics.LogEvent("level_complete",
            new Parameter("level_number", level),
            new Parameter("duration_seconds", duration));
    }

    public static void TrackGameOver(string cause, int score)
    {
        FirebaseAnalytics.LogEvent("game_over",
            new Parameter("cause", cause),
            new Parameter("score", score));
    }
}
```

**출시 후 모니터링 지표**:

| 지표 | 목표 기준 | 의미 |
|---|---|---|
| D1 리텐션 | 30% 이상 | 첫 날 재방문율 |
| D7 리텐션 | 10% 이상 | 일주일 후 재방문율 |
| 세션당 플레이 시간 | 계획 세션의 100% | 충분히 즐기고 있음 |
| 크래시율 | 0.5% 미만 | 안정성 |
| 1~5스타 비율 | 4.0 이상 평균 | 유저 만족도 |

## 출시 후 업데이트 전략

### 데이터 기반 업데이트 결정

```
우선순위 결정 프레임워크:

1. 크래시/버그 수정 → 즉시
2. 리텐션 저하 원인 → 2주 이내
3. 유저 요청 상위 기능 → 1개월 이내
4. 신규 콘텐츠 → 장기 로드맵

업데이트 주기:
- 핫픽스: 즉시 (크래시, 진행 불가)
- 패치: 2주 (밸런스, 마이너 버그)
- 콘텐츠 업데이트: 4~8주
```

### 커뮤니티 피드백 수집

```
채널:
- 스토어 리뷰 모니터링 (매일)
- 커뮤니티 포럼/Discord
- 소셜 미디어 멘션

응대 원칙:
- 버그 리포트는 24시간 내 응대
- 부정적 리뷰는 공손하게 대응
- 수정 예정 항목은 공개 로드맵으로 공유
```
