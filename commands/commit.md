다음 순서대로 커밋을 수행하라.

## 1. 변경 사항 파악

아래 명령을 실행하여 변경 내용을 파악하라.

```bash
git status --short
git diff HEAD
git diff --staged
```

## 2. 컨텍스트 파악

아래 파일이 존재하면 읽어 작업 배경을 파악하라.

- `work-log.md` — 최근 완료 작업
- `designer-log.md` — 마지막 항목 (기획 맥락)
- `developer-log.md` — 마지막 항목 (구현 맥락)

## 3. 커밋 타입 결정

변경 파일을 분석하여 아래 중 하나를 선택하라. 여러 타입에 걸치면 가장 핵심적인 변경 기준으로 선택.

| 타입 | 조건 |
|------|------|
| `feat` | 새 기능·스킬·에이전트 추가 |
| `fix` | 버그·오류 수정 |
| `refactor` | 동작 변경 없는 구조 개선 |
| `docs` | 문서만 변경 |
| `test` | 테스트 코드만 변경 |
| `chore` | 설정·버전·패키지 변경 |
| `skill` | 스킬 파일만 변경 |
| `agent` | 에이전트 파일만 변경 |

## 4. 파일 스테이징

스테이징되지 않은 관련 파일을 스테이징하라. `.env`·인증 정보 등 민감한 파일이 포함된 경우 커밋 전 사용자에게 확인. 변경 사항이 없으면 커밋하지 말고 사용자에게 알림.

```bash
git add [변경된 파일들]
```

## 5. 커밋 메시지 작성 후 커밋

아래 포맷으로 커밋 메시지를 작성하여 커밋하라.

```

## 배경

[왜 이 작업을 했는지 — 요구사항, 해결하려는 문제, 이전 상태의 한계]

## 변경 내용

[파일/모듈명]: [무엇을 어떻게 변경했는지]
[파일/모듈명]: [무엇을 어떻게 변경했는지]

## 아키텍처 결정 (해당 시)

## 다음 작업 (해당 시)

[이어서 해야 할 작업 — 없으면 섹션 생략]
```
