요구사항: $ARGUMENTS

위 요구사항을 아래 5단계로 구현하라. **각 단계가 완료되면 반드시 결과를 사용자에게 보여주고 피드백을 받은 뒤 다음 단계로 진행하라.** 사용자가 수정을 요청하면 해당 단계를 재수행하고 다시 피드백을 받아라.

---

## 1단계 — 계획 (pm-planner)

요구사항을 분석하여 계획서를 작성하라.

1. 요구사항에서 slug를 결정하라 (영문 소문자·하이픈, 예: `score-system`)
2. pm-planner 스킬에 따라 `[slug].plan.md`를 작성하라:
   - 개요
   - 핵심 기능 목록
   - 기술 스택
   - 상세 작업 목록 (체크리스트)
   - 성공 지표
3. 작성된 계획서 전체를 사용자에게 보여주고 아래를 물어라:

> **[1단계 완료] 계획서를 검토해 주세요.**
> 수정이 필요하면 말씀해 주세요. 진행해도 되면 "계속" 또는 "다음"이라고 해주세요.

사용자가 수정을 요청하면 계획서를 수정하고 다시 보여준 뒤 확인을 기다려라.

---

## 2단계 — 기획 (pm-designer)

계획서를 바탕으로 구현 설계서를 작성하라.

1. `[slug].plan.md`를 읽어라
2. `references/project-context.md`를 확인하라:
   - 있으면 읽어 기술 스택·패턴·코딩 스타일을 파악하라
   - 없으면 project-context 스킬을 실행하여 먼저 생성하라
3. 기존 관련 코드가 있으면 Glob·Grep·Read로 확인하라
4. pm-designer 스킬에 따라 `[slug].design.md`를 작성하라:
   - 계획 유형(신기능)에 맞게 시스템 아키텍처·데이터 스키마·클래스·메서드 설계
   - 체크리스트 없이 확정된 명세로 서술
5. 작성된 설계서 전체를 사용자에게 보여주고 아래를 물어라:

> **[2단계 완료] 설계서를 검토해 주세요.**
> 수정이 필요하면 말씀해 주세요. 진행해도 되면 "계속" 또는 "다음"이라고 해주세요.

사용자가 수정을 요청하면 설계서를 수정하고 다시 보여준 뒤 확인을 기다려라.

---

## 3단계 — 개발 (developer)

설계서에 따라 TDD로 구현하라.

1. `[slug].design.md`를 읽어라
2. `references/project-context.md`를 읽어 코딩 스타일·패턴을 파악하라
3. dev-tdd 스킬과 dev-lightweight-ddd 스킬에 따라 구현하라:
   - 설계서의 각 항목을 Red → Green → Refactor 사이클로 구현
   - 한 번에 하나의 테스트만 작성하고 통과시킨 뒤 다음으로 진행
   - IDE 수동 작업이 필요하면 setup-guide 스킬로 가이드를 작성하고 사용자 완료 후 setup-validator로 검증
4. 구현이 완료되면 생성·수정된 파일 목록과 각 파일의 변경 내용을 요약하여 사용자에게 보여주고 아래를 물어라:

> **[3단계 완료] 구현 내용을 검토해 주세요.**
> 수정이 필요하면 말씀해 주세요. 진행해도 되면 "계속" 또는 "다음"이라고 해주세요.

사용자가 수정을 요청하면 해당 부분을 수정하고 다시 요약을 보여준 뒤 확인을 기다려라.

---

## 4단계 — 검수 (gap-detector)

설계서와 실제 구현을 대조하여 갭을 분석하라.

1. `[slug].plan.md`와 `[slug].design.md`를 읽어라
2. Glob·Grep·Read로 구현된 코드를 직접 확인하라
3. gap-detector 스킬에 따라 `[slug].gap.md`를 작성하라:
   - 각 항목을 ✅ / ⚠️ / ❌ 로 판정
   - 일치율(%) 및 등급 산출
4. 갭 분석 결과를 사용자에게 보여주고 아래를 물어라:

> **[4단계 완료] 검수 결과를 확인해 주세요.**
> ❌·⚠️ 항목이 있으면 수정 후 재검수하겠습니다. 진행해도 되면 "계속" 또는 "다음"이라고 해주세요.

- ❌·⚠️ 항목이 있고 사용자가 수정을 요청하면 3단계로 돌아가 해당 항목을 수정한 뒤 갭 분석을 재실행하라
- 사용자가 현재 상태로 진행을 승인하면 다음 단계로 넘어가라

---

## 5단계 — 커밋 (commit)

작업 내용을 구조화된 커밋 메시지로 커밋하라.

1. `git status --short`, `git diff HEAD`로 변경 사항을 파악하라
2. `work-log.md`, `developer-log.md`가 있으면 읽어 맥락을 파악하라
3. 아래 포맷으로 커밋 메시지를 작성하라:

```
feat: [요구사항 한 줄 요약]

## 배경

[요구사항과 이 기능이 필요한 이유]

## 변경 내용

[파일/모듈명]: [무엇을 어떻게 변경했는지]
[파일/모듈명]: [무엇을 어떻게 변경했는지]

## 아키텍처 결정 (해당 시)

[설계 선택 이유 — 없으면 섹션 생략]

## 다음 작업 (해당 시)

[이어서 해야 할 작업 — 없으면 섹션 생략]
```

4. 커밋 메시지를 사용자에게 보여주고 아래를 물어라:

> **[5단계 완료 전] 커밋 메시지를 확인해 주세요.**
> 수정이 필요하면 말씀해 주세요. 진행해도 되면 "커밋" 또는 "확인"이라고 해주세요.

5. 사용자 승인 후 커밋하라:

```bash
git add [변경된 파일들]
git commit -m "$(cat <<'EOF'
[커밋 메시지]
EOF
)"
```

6. 커밋 완료 후 아래를 출력하라:

> **[완료] `[slug]` 기능 구현이 완료되었습니다.**
> 산출물: `[slug].plan.md` · `[slug].design.md` · `[slug].gap.md`
