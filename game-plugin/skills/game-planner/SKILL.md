# Game Planner - MVP 기반 게임 개발 계획 수립

## 목적

게임 분석 결과 또는 게임 아이디어를 입력받아, **MVP 방법론**에 따라 단계별로 실행 가능한 개발 계획을 수립합니다.

MVP의 핵심 원칙: **가장 빠르게 핵심 재미를 검증**하고, 검증된 재미 위에 점진적으로 콘텐츠와 완성도를 쌓아 올립니다.

## 입력 형태

다음 중 하나를 입력받아 계획을 수립합니다:

- **게임 분석 결과**: `game-analysis` 스킬이 작성한 레퍼런스 게임 분석 문서
- **게임 아이디어**: 만들고 싶은 게임 설명 (장르, 핵심 메카닉, 타겟 유저 등)
- **두 가지 조합**: 아이디어 + 참고할 레퍼런스 게임 분석

입력이 부족하면 다음을 질문합니다:
1. 팀 규모 (솔로 / 2~3인 / 5인 이상)
2. 전체 개발 기간 목표
3. 핵심 재미 요소 (한 문장으로)
4. 타겟 플랫폼 (PC, 모바일, 콘솔)

## 계획 수립 워크플로우

### 1단계: 핵심 재미(Core Fun) 정의

모든 계획의 출발점. MVP가 반드시 증명해야 하는 단 하나의 경험.

**질문**:
- 이 게임에서 플레이어가 "재밌다"고 느끼는 순간은 언제인가?
- 10초 안에 설명할 수 있는가?
- 이 재미를 느끼기 위해 최소한 무엇이 필요한가?

**예시**:
```
로그라이크 서바이벌:
"몰려오는 적을 자동 공격으로 물리치며 레벨업할 때마다
스킬을 조합하는 재미"

덱빌딩 카드게임:
"카드 조합으로 시너지를 만들어 강력한 한 방을 터뜨리는 재미"

타워디펜스:
"배치 전략으로 밀려오는 적을 막아내는 퍼즐 같은 재미"
```

### 2단계: MVP 범위 산정

핵심 재미를 검증하기 위한 **최소한의 기능 목록**을 정의합니다.

**MVP 포함 기준**:
- ✅ 핵심 재미를 직접 구성하는 요소
- ✅ 핵심 재미를 느끼기 위해 반드시 필요한 요소
- ❌ "있으면 좋은" 요소 (이후 단계로)
- ❌ 콘텐츠 다양성 (종류가 많을 필요 없음)
- ❌ 완성도 (그래픽, 사운드, 애니메이션)
- ❌ 메타 시스템 (세이브, 업적, 랭킹)

**MVP 범위 산정 체크리스트**:
```
□ 플레이어가 핵심 액션을 1회 수행할 수 있는가?
□ 그 액션에 의미 있는 피드백이 있는가?
□ 최소 1개의 승리/실패 조건이 있는가?
□ 재시작이 가능한가?
□ 핵심 루프를 3회 이상 반복할 수 있는가?
```

### 3단계: 단계별 계획 구조화

MVP → 알파 → 베타 → 출시 준비 단계로 구조화합니다.

각 단계는 **이전 단계 위에 쌓이며**, 각 단계 완료 시 플레이 가능한 빌드가 존재합니다.

### 4단계: 작업 항목 도출 및 우선순위 지정

각 단계의 기능을 구현 작업으로 분해하고 의존성에 따라 순서를 정합니다.

### 5단계: 리스크 식별 및 대응 계획

개발 중 발생할 수 있는 기술적·디자인 리스크를 미리 식별합니다.

## 모범 사례

1. **핵심 재미 먼저**: MVP 범위는 항상 핵심 재미로부터 역방향으로 정의
2. **작게 시작**: 처음 생각한 MVP 범위의 절반만 해도 충분한 경우가 많음
3. **플레이 테스트 조기 실시**: MVP 완료 즉시 외부인에게 플레이 테스트
4. **데이터 기반 확장**: 재미가 검증된 요소만 확장, 가정으로 콘텐츠 추가 금지
5. **기술 부채 경계**: MVP에서 빠른 구현은 허용하되 알파 전에 리팩토링
6. **범위 고정**: 알파 이후 신규 핵심 메카닉 추가는 새 프로젝트 수준의 결정
7. **현실적인 일정**: 예상 기간 × 1.5를 실제 일정으로 사용
